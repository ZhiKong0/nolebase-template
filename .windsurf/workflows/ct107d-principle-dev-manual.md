# 蓝桥杯 CT107D 单片机原理及开发手册（Workflow）

本 workflow 用于让 AI 在进行 CT107D（STC15F2K60S2）开发时，能够**按统一的硬件/总线抽象**理解原理图连接与运行原理，并将其映射到代码实现（System/Hardware/User 分层或单文件工程均适用）。

## 0. 适用范围与基本约束

- 主控：`STC15F2K60S2`
- 开发板：`CT107D`
- 关键器件：`74HC138 + 74HC573 + ULN2003 + 8位共阳数码管 + 8LED + 4x4矩阵键盘`
- 软件侧常见抽象：
  - `BusWrite(addr, dat)`：通过 P2 选通锁存器、P0 输出数据、再关闭锁存
  - 数码管动态扫描：Timer 中断 1ms（或更快）周期调用 `Nixie_ScanOnce()`
  - 按键扫描：矩阵键盘逐列扫描 + 软件消抖 + 边沿检测

## 1. 板级总线架构（最重要的核心）

CT107D 的“显示/LED/蜂鸣器/继电器”等外设并非直接连到 MCU 的独立 GPIO，而是通过：

- **P0：8位数据总线**（D0~D7）
- **P2：地址/译码控制**（主要用 P2.4~P2.7）
- **74HC138：3-8译码**（选择哪一路外设锁存器）
- **74HC573：8位锁存器**（锁存 P0 输出，保持外设状态）

### 1.1 74HC138 译码器（U17）

- 使能：
  - `G1 = VCC`（恒为 1）
  - `G2A = P2^7`（低有效，文档中常称 WR 低电平使能）
  - `G2B = GND`（恒为 0）
- 选择输入（CBA）：`P2^6 P2^5 P2^4`

当 `P2^7=0` 且 CBA 为不同组合时，会把 Y4~Y7 中某一路拉低（选通对应锁存器）。

### 1.2 外设选通地址（P2 写入值）

以下地址是你写入 `P2` 的值（本质是把 `P2.7=0` + `P2.6~P2.4=CBA` 组合好）：

- `0x80`：`CBA=100` -> `Y4` -> **LED锁存器**
- `0xA0`：`CBA=101` -> `Y5` -> **ULN2003锁存器（蜂鸣器/继电器）**
- `0xC0`：`CBA=110` -> `Y6` -> **数码管位选锁存器**
- `0xE0`：`CBA=111` -> `Y7` -> **数码管段选锁存器**

建议在代码里统一用宏：

- `LATCH_LED = 0x80`
- `LATCH_ULN = 0xA0`
- `LATCH_DIG = 0xC0`
- `LATCH_SEG = 0xE0`

### 1.3 BusWrite 的硬件时序

**正确的写总线顺序：**

1. `P2 = addr;` 选通对应锁存器（74HC573 的 LE 有效）
2. `P0 = dat;` 输出数据
3. `_nop_();` 短暂延时（保证数据建立/保持）
4. `P2 = 0x00;` 关闭选通，锁存数据

这一步的本质：**外设状态由 74HC573 的 Q 端保持**，不会因为 P0 后续变化而改变。

## 2. 外设模块原理与代码映射

### 2.1 LED（8个）

- 连接：LED 通过 74HC573（Y4 选通）输出控制。
- 逻辑：**低电平点亮**（常见现象：写 `0xFE` 只亮 L1）。
- 代码：`BusWrite(LATCH_LED, led_dat)`

### 2.2 数码管（8位共阳）

- 结构：
  - 位选：Y6 锁存器，**高电平选通某位 COMx**
  - 段选：Y7 锁存器，**低电平点亮段 a~g、dp**
- 动态扫描：
  - 每 1ms（或更快）刷新一位，一轮 8 位 = 8ms，刷新率约 125Hz（肉眼稳定）
- 消影（防重影/鬼影）：
  - 先关位选、关段选，再开新位选、输出段码

常见段码（共阳，低亮）：
- 0：`0xC0`
- 1：`0xF9`
- 2：`0xA4`
- 3：`0xB0`
- 4：`0x99`
- 5：`0x92`
- 6：`0x82`
- 7：`0xF8`
- 8：`0x80`
- 9：`0x90`
- 灭：`0xFF`

### 2.3 蜂鸣器 / 继电器（ULN2003）

- 选通：`0xA0`（Y5）
- 驱动：ULN2003 达林顿阵列，**输入高电平时输出为低电平有效**（外设动作）
- 位定义（来自手册整理）：
  - `P0.6`（bit6）控制蜂鸣器
  - `P0.4`（bit4）控制继电器

因此常见写法：
- 蜂鸣器开：写入数据 bit6=1
- 继电器开：写入数据 bit4=1
- 全关：`0x00`

### 2.4 矩阵键盘（4x4=16键）

- 行线：`P3.0 ~ P3.3`
- 列线：`P4.4`（列0）、`P4.2`（列1）、`P3.5`（列2）、`P3.4`（列3）

逐列扫描法（推荐的稳定做法）：
1. 行线配置为输入（上拉），默认读到 1
2. 依次让某一列输出 0（其余列输出 1）
3. 读取行线，若某行读到 0，表示该行该列按下
4. key 编码：`key = row*4 + col + 1`（范围 1~16）

消抖建议：
- 不要只靠边沿检测；边沿检测只能“判断变化”，不能保证变化真实稳定
- 推荐“逐位独立消抖 + 边沿检测”的 Key 模块

## 3. 跳线帽（务必检查，否则“代码没问题但硬件不响应”）

- `J2`：数码管位选
  - 1-2 短接：使用板载数码管
- `J5`：键盘模式
  - 1-2 短接：矩阵键盘
  - 2-3 短接：独立按键
- `J13`：温度传感器模式
  - 1-2 短接：DS18B20
  - 2-3 短接：AD/DA

## 4. 引脚速查（做驱动前先对照）

- **P0**：数据总线（LED/位选/段选/ULN 等均通过锁存）
- **P2.4~P2.7**：138 译码选择 + 使能
- **I2C**：`P2.0=SCL`，`P2.1=SDA`
- **DS18B20**：`P1.4=DQ`
- **矩阵键盘**：行 `P3.0~P3.3`，列 `P4.4/P4.2/P3.5/P3.4`

## 5. 系统初始化的“安全默认态”

上电后推荐顺序（避免乱闪、乱叫）：

1. IO 全部拉高（避免浮空）
2. 关 LED（写 `0xFF` 到 LED 锁存器）
3. 关数码管位选（写 `0x00` 到 DIG 锁存器）
4. 关数码管段选（写 `0xFF` 到 SEG 锁存器）
5. 关蜂鸣器/继电器（写 `0x00` 到 ULN 锁存器）

## 6. 常见故障定位（从硬件原理出发）

- 数码管不亮/乱亮：
  - 检查 `J2` 跳线
  - 确认 Timer 中断持续调用扫描函数（扫描频率足够）
  - 确认“先消影再输出”
- LED 全反：
  - CT107D 的 LED 是**低电平亮**，注意取反
- 蜂鸣器/继电器不受控：
  - 确认写入的是 `0xA0`（ULN）而不是 LED/DIG/SEG
  - 注意 ULN 是“输入高 -> 输出低有效”
- 按键不灵：
  - 检查 `J5` 模式
  - 矩阵键盘扫描列线的端口方向设置是否正确
  - 加软件消抖（逐位独立，主循环高速或 1ms 定时）

## 7. AI 在 CT107D 工程中的工作方式（强制遵循）

当你要求 AI 编写/修复 CT107D 工程时，AI 必须：

1. **先确认硬件抽象**：使用 `BusWrite` 对应 4 个锁存地址
2. **任何显示相关逻辑**都必须保证：Timer 扫描 + 消影
3. **任何按键相关逻辑**都必须保证：扫描 + 消抖 + 边沿
4. 若现象与代码不符，优先检查：跳线帽/锁存地址/高低有效逻辑

## 8. AI开发规范（核心原则：绝不修改库函数）

### 8.1 不可修改的文件/文件夹

```
以下文件夹中的文件绝对禁止修改：
├── System/           # 系统级库函数
│   ├── Bus.c/h       # 总线操作
│   ├── Init.c/h      # 系统初始化
│   └── Public.h      # 公共定义
│
├── Hardware/         # 硬件驱动库
│   ├── LED.c/h       # LED驱动
│   ├── Nixie.c/h     # 数码管驱动
│   ├── Key.c/h       # 按键驱动
│   └── ...
│
└── Library/          # 第三方库文件
    └── ...
```

### 8.2 必须使用库函数进行开发

**正确的开发方式：**
```c
// ✅ 正确：调用库函数实现功能
#include "System/Bus.h"
#include "Hardware/Nixie.h"

void main(void) {
    System_Init();          // 调用系统初始化
    Nixie_ClearAll();       // 调用数码管清零
    
    while(1) {
        BusWrite(LATCH_LED, 0xAA);   // 使用库函数控制LED
        Nixie_Display(1234);          // 使用库函数显示数字
    }
}
```

**错误的开发方式：**
```c
// ❌ 错误：直接修改库函数或复制粘贴到自己文件
void My_System_Init(void) {     // 不要自己重写系统初始化
    P0 = 0xFF;
    P2 = 0x80;
    // ... 复制库函数内容
}

// ❌ 错误：直接操作P2口而不通过BusWrite
P2 = 0x80;   // 不要这样直接操作
P0 = 0xAA;   // 应该使用 BusWrite(0x80, 0xAA)
```

### 8.3 库函数冲突解决方法

**问题：函数名冲突**
```
Error: multiple definition of `System_Init'
```

**解决方案：**
```c
// 如果库函数名冲突，使用重命名自己的函数
void My_System_Init(void) {  // ✅ 加上My_前缀区分
    // 你的实现
}

// 或者使用static限制作用域
static void Local_Init(void) {  // ✅ 使用static避免链接冲突
    // 你的实现
}
```

### 8.4 允许修改的文件

**可以修改的文件：**
- `User/main.c` - 主程序文件（用户代码入口）
- `User/*.c` - 用户自己创建的.c文件
- 项目根目录下的工程文件

### 8.5 开发规范

#### 1. 头文件包含
```c
// ✅ 正确：必须包含必要的库头文件
#include "System/Bus.h"
#include "Hardware/LED.h"
#include "Hardware/Nixie.h"

// 不要重复定义库中已有的内容
// ❌ 错误：不要重新定义 LATCH_LED 等宏
#define LATCH_LED 0x80   // 库文件中已有，不要重复定义
```

#### 2. 变量命名
```c
// ✅ 正确：使用有意义的变量名
u8 key_value;           // 按键值
u16 countdown_value;    // 倒计时数值

// 避免与库函数同名
u8 key;     // 不好，可能与库变量冲突
```

#### 3. 函数调用
```c
// ✅ 正确：调用库函数
void Timer0_ISR(void) interrupt 1 {
    Nixie_ScanOnce();   // 调用库函数扫描数码管
}

// ❌ 错误：不要自己实现扫描逻辑
void My_Nixie_Scan(void) {
    // 不要复制库函数的扫描代码到这里
}
```

### 8.6 常见问题处理

**Q1: 库函数不够用怎么办？**
A: 在User文件夹创建自己的函数文件，不要修改原有库
```
User/
├── main.c              # 主程序
├── My_Functions.c     # 你的功能函数
└── My_Functions.h     # 头文件
```

**Q2: 库函数有bug怎么办？**
A: 报告给用户，由用户决定是否修改，不要擅自修改

**Q3: 不确定某个函数是否在库中？**
A: 查看System/和Hardware/文件夹，或使用grep搜索

### 8.7 开发流程建议

```
1. 理解需求
   ↓
2. 查看库函数文档（MCU_Tools/中的手册）
   ↓
3. 规划使用哪些库函数
   ↓
4. 在main.c或User文件夹实现业务逻辑
   ↓
5. 编译测试（不要修改库）
   ↓
6. 如有问题，先检查自己的调用方式，而非修改库
```

### 8.8 重要提醒

**记住这个原则：**
- 库函数是基础设施，就像房子的地基
- 你不应该在地基上打洞，而是在地基上盖房子
- 如果地基有问题，应该让建地基的人来修，而不是你自己动手

**开发时反复问自己：**
1. 我修改的代码在System/或Hardware/文件夹吗？
2. 如果是 → 停止修改！
3. 如果不是 → 继续开发

### 8.9 违反规范的后果

```
❌ 修改库函数可能导致：
├── 其他代码依赖关系被破坏
├── 编译链接错误
├── 硬件功能异常
├── 项目无法复用
└── 后续维护困难

✅ 遵守规范的好处：
├── 代码模块化清晰
├── 便于调试和维护
├── 项目可移植性强
├── 减少不必要的问题
└── 符合蓝桥杯竞赛要求
```
